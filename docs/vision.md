# FAB System — 프로젝트 비전

## 핵심 목표

**수동 Planning Bridge → WMS → MES를 실제 계획·실행 흐름으로 연결하는 제조 운영 플랫폼을 구축한다.**

```text
Planning Bridge (수동 계획·조달 기준)
  ↕
WMS (창고·입출고·위치·로트)
  ↕
MES (작업지시·피킹·공정 실적)
```

## 현재 상태

| 레이어 | 역할 | 현황 |
|---|---|---|
| WMS | 입출고, 위치, 로트·용기 재고 추적 | ✅ Phase 1 구현 |
| Planning Bridge/SCM | 자재·공급사 기준, 수동 입고계획, WMS 실입고 연결 | 🔄 입고계획 Phase 1, 생산계획 연결 필요 |
| MES | 작업지시, BOM, FEFO 피킹, 공정 준비도 | ✅ Phase 1 구현 |

자재 ID를 운영 정보의 기준축으로 사용한다. 자재 운영 허브에서 전체·창고별 재고, Lot 품질, 공정 사용량, BOM, 승인 공급사와 리드타임을 하나의 흐름으로 조회하고, 조달 기준 수정은 SCM과 What-if에 동일하게 반영된다. 공정별 사용량을 일사용량과 DOH의 기준으로 사용하고, Planning Bridge의 확정 입고계획은 WMS 실입고와 연결되며 WMS 로트 재고는 MES 피킹과 movement 기록으로 이어진다. 다음 핵심 과제는 수동 생산계획과 MES 공정 실적·원가를 같은 흐름에 연결하는 것이다.

## 디지털 트윈 방향

과거의 랜덤 자동운전과 타임 액셀러레이터는 현행 목표에서 제외한다. 디지털 트윈은 실제 운영 스냅샷에 사용자가 명시한 조건을 적용하는 **설명 가능하고 재현 가능한 What-if** 도구로 발전시킨다.

- 수요 변화, 입고 계획, 공급 지연 조건을 명시적으로 입력
- 동일한 입력은 동일한 결과를 반환
- 결과를 `SIMULATION`으로 표시해 운영 데이터와 구분
- 실제 반영 전에 재고, 결품, 창고 Capacity 영향을 비교
- 입력, 결과, 검토 및 승인 이력을 보존
- 통상 리드타임과 보수적 최대 리드타임을 분리해 발주 가능 범위를 설명

## 로드맵

### Phase 1 — WMS·MES 연결 기반 ✅

- 위치·로트·용기 기반 WMS와 FEFO 출고
- BOM 기반 MES 작업지시와 자재 피킹
- 공정 사용량에서 재고 DOH까지 단일 계산 흐름

### Phase 2 — Planning Bridge 수동 계획·실행 연결

- 승인 공급사와 주/보조 조달 역할 관리
- 재고 목록에서 진입하는 자재 단위 종합 운영 허브
- 최소·기준·최대·현재 예상 리드타임 관리
- 수동 입고계획의 초안·확정·취소 워크플로우
- 부분입고·잔여량·완료 상태와 WMS 실입고 반영
- 공급사, 리드타임, 단가 및 재고 평가 연결

Planning Bridge는 외부 ERP를 대체하지 않는다. 회계·세무·결산 또는 외부 거래를 처리하지 않으며, 사용자가 입력한 계획과 WMS·MES 실행의 연결만 책임진다. 입력값의 정확성과 외부 ERP 자료와의 일치 여부는 사용자가 확인한다.

### Phase 3 — MES 실적 고도화

- 공정 실적·수율·불량 기록
- WMS 피킹과 실제 투입·반납 연결
- 외부 원가 자료와 생산 실적의 수동 반영

### Phase 4 — 설명 가능한 디지털 트윈

- 실공장 스냅샷과 What-if 결과 병렬 비교
- 수요·입고·지연 조건 기반 의사결정 지원
- 시나리오의 입력·결과·승인 이력 보존

### 실제 fab과의 차이점
  - 실제 fab은 3~4개월의 기간이 필요하지만, 우리 시스템은 10~11분 이내에 완료되므로 1.2~1.6만배 정도 차이

### "타임 액셀러레이터 제외"와의 관계

25번 줄의 "과거의 랜덤 자동운전과 타임 액셀러레이터는 현행 목표에서 제외한다"는 **What-if 디지털 트윈(Phase 4)**, 즉 결과가 재현 가능해야 하고 입력·결과·승인 이력이 남아야 하는 의사결정 시뮬레이션에 해당하는 원칙이다.

FOUP 3D 실시간 추적(`waferLots`/`waferLotStepEvents` 실행 원장, `AUTO_ADVANCE_INTERVAL_MS` 자동 진행)은 이것과 다른 층위다 — 랜덤이 아니라 `routeMaster`에 정의된 순서를 결정론적으로 따라가고, 각 스텝이 실제 이벤트(`OPERATOR_CONFIRM` 또는 `MES_TELEMETRY`)로 원장에 기록되며, 패키징 진입 시 실제 M20 파일럿 워크오더·발주 에이전트를 트리거한다. 즉 "설명 불가능한 배속 연출"이 아니라 "MES 텔레메트리가 아직 없어서 사람이 지켜보지 않아도 흐름이 계속 돌아가게 하는 임시 대역"이며, 실제 설비 신호가 붙으면 `MES_TELEMETRY` 트리거가 그대로 대체한다.
