# 트리오 기획 세션 — 운영 관리 데이터 입력 구조

**날짜:** 2026-07-12  
**기능 요청:** FAB 시스템의 실제 운영 데이터를 입력·관리하는 새 탭을 어떤 구조로 만들지 설계  
**의제:** 운영 데이터 관리, 재고·입고·생산계획 입력, 시뮬레이션과의 경계

---

## 문제 정의

현재 FAB 시스템에는 재고, 공정 사용량, 창고, 위험, 입고 및 시장 시뮬레이션 화면이 존재한다. 그러나 이 화면들의 기준이 되는 실제 운영 데이터를 사용자가 안전하게 입력·확정·수정하는 영역이 없다.

이번 세션에서는 단순한 데이터 입력 폼이 아니라, 프로젝트의 현재 상태를 관리하는 **운영 관리 영역**을 신설하는 방안을 논의했다.

---

## 크리

저는 이걸 단순한 “데이터 입력 탭”이 아니라 **프로젝트의 현실을 관리하는 운영 관제실**로 정의하고 싶어요.

사용자가 빈 폼을 열고 DB 항목을 찾아다니는 대신, 시스템이 오늘 처리해야 할 일을 먼저 보여주는 거죠.

- 오늘 입고 예정
- 재고 조정이 필요한 항목
- 생산·수요 계획 변경
- 창고 용량과 위치 변경
- 입력 오류와 미확인 데이터

저는 다음 세 가지 개념을 고려했어요.

### 1. 운영 콘솔

`운영 관리`를 새 탭으로 두고, 첫 화면에서 오늘 관리해야 할 데이터를 보여줍니다. 사용자는 메뉴를 찾는 대신 “오늘 할 일”을 선택해 입력합니다.

### 2. 데이터 허브

데이터를 성격별로 나눕니다.

- 마스터 데이터: 자재, 공정, 공급사, 창고, 단위, 안전재고
- 운영 데이터: 현재 재고, 예정 입고, 생산계획, 사용량, 창고 점유
- 데이터 품질: 누락, 오래된 값, 단위 불일치, 연결되지 않은 자재
- 변경 이력: 누가 언제 무엇을 바꾸었는지

### 3. 상황판형 입력

시스템이 입력이 필요한 상황을 먼저 요청합니다.

- “불산 재고가 3일간 갱신되지 않았습니다” → `재고 입력`
- “7월 20일 입고 수량이 미확정입니다” → `입고 확정`
- “HBM 생산계획이 지난달 값입니다” → `계획 갱신`
- “WH-C 용량 정보가 없습니다” → `용량 등록`

제 추천은 **운영 콘솔을 뼈대로 만들고 상황판형 입력 경험을 섞는 것**입니다.

가장 중요한 구분은 운영 데이터와 시나리오 데이터를 섞지 않는 것입니다.

- 운영 관리에서 저장 → 실제 대시보드 기준값 변경
- 시뮬레이션에서 저장 → 가상 시나리오에만 반영
- 시나리오를 운영에 적용 → 검토·승인 후 실제 계획으로 전환

제가 생각한 초기 구조는 이렇습니다.

```text
운영 관리
├─ 오늘의 운영 현황
│  ├─ 입력 필요
│  ├─ 확인 필요
│  └─ 오늘 반영된 변경
├─ 재고 및 실사
├─ 입고 계획
├─ 생산·수요 계획
├─ 창고 운영
├─ 기준정보
└─ 변경 이력
```

## 패브

저는 크리의 “운영 관제실” 정의에 동의합니다. 다만 현재 단계에서 메뉴 전체를 한번에 만드는 것은 과합니다.

MVP는 `/operations` 아래 다섯 영역으로 제한하는 것이 맞습니다.

1. 오늘의 운영
2. 입고 계획
3. 생산 계획
4. 재고 실사·조정
5. 변경 이력

기준정보와 창고 구조 관리는 2차로 미루는 편이 좋습니다.

### 재고는 직접 덮어쓰지 않는다

가장 중요한 수정점은 `Inventory.quantity`를 사용자가 직접 덮어쓰게 하면 안 된다는 것입니다.

수량의 원장은 입고·출고·실사조정 이벤트이고, `Inventory`는 이 이벤트를 집계한 현재 잔액이어야 합니다.

```text
장부 재고  8,000kg
실사 재고  7,850kg
차이          -150kg
사유          계측 오차 및 잔량 폐기
```

이 경우 `7,850kg`으로 덮어쓰는 것이 아니라 `-150kg 조정`을 기록한 후 재고에 반영해야 합니다.

### 입고 계획과 실제 입고를 분리한다

입고 계획은 다음 상태를 갖습니다.

```text
DRAFT → CONFIRMED → RECEIVED
                   └→ CANCELLED
```

- `DRAFT`: 대시보드와 시뮬레이션 기준선에서 제외
- `CONFIRMED`: 예정 입고로 미래 계산에 반영
- `RECEIVED`: 실제 수량과 시각을 확인한 후 재고 및 이동 기록 생성
- `CANCELLED`: 삭제하지 않고 취소 사유 기록

### 생산계획과 자재 원단위를 분리한다

현재 `ProcessUsage.monthlyQty`를 생산계획과 겸용하면 안 됩니다.

- `ProcessUsage`: 특정 제품을 생산할 때 필요한 자재 원단위
- `ProductionPlan`: 특정 기간에 실제로 생산할 제품 수량

```text
미래 자재 사용량
= 생산계획량 × 제품·공정·자재별 원단위
```

### 저장 규칙

1. 초안은 계산에 반영하지 않고, 확정된 데이터만 반영한다.
2. 현재 재고를 직접 수정하지 않고 입고·출고·실사조정 기록을 통해 변경한다.
3. 미래 계획은 덮어쓰지 않고 새 버전을 발행한다.
4. 현장 발생 시각과 시스템 입력 시각을 분리한다.
5. 자재 단위는 서버에서 변환·검증한다.
6. 시나리오 결과는 운영 컬렉션에 직접 쓰지 않는다.

## 엑스

저는 사이드바에 **`운영 관리`**라는 상위 탭을 추가하겠습니다.

```text
운영 관리
├─ 오늘의 운영
├─ 입고 계획
├─ 생산 계획
├─ 재고 실사·조정
└─ 변경 이력
```

공급업체, 자재, 공정별 원단위, 창고 용량 같은 저빈도 설정은 초기 운영 화면에 섞지 않고, 향후 `기준정보 관리`로 분리하겠습니다.

### 첫 화면 — 오늘의 운영

```text
┌────────────────────────────────────────────────────┐
│ 운영 관리 · 2026.07.12              데이터 기준 14:32  │
│ 오늘 운영 데이터 3건의 확인이 필요합니다.              │
├────────────────────────────────────────────────────┤
│ [입고 등록]       [생산계획 등록]      [재고 조정]     │
│ 오늘 예정 7건     미입력 계획 2건       실사 차이 3건   │
├────────────────────────────────────────────────────┤
│ 확인 대기                                             │
│ ! 불산 입고 예정시간 경과       [상세 확인] [입고 처리] │
│ ! HBM 다음 주 계획 미확정       [계획 입력]             │
│ ! WH-C 실사 차이 +120kg         [검토]                 │
├────────────────────────────────────────────────────┤
│ 최근 변경                                             │
│ 14:28 김OO · 불산 입고 5,000kg 확정                    │
│ 13:52 이OO · WH-C 재고 +120kg 조정 요청                │
└────────────────────────────────────────────────────┘
```

### 입고 계획 흐름

```text
초안 작성 → 검토 → 입고 확정 → 실제 입고 처리
 DRAFT       DRAFT   CONFIRMED      RECEIVED
```

실제 입고 처리 시 예정 수량을 미리 채우되, 사용자가 실제 수량, 시간, 로트와 적재 위치를 다시 확인하게 합니다.

### 생산 계획 흐름

```text
기간 선택
→ 제품별 생산량 입력
→ 자재 소요량 미리보기
→ 재고·창고·위험 영향 확인
→ 초안 저장 또는 확정
```

확정 전에 입력한 생산계획이 만들 영향을 바로 보여줍니다.

> HBM 계획을 10,000 → 13,000 lot로 변경하면  
> 불산 일사용량 +480kg, EUV PR +32L가 예상됩니다.  
> 현재 입고계획 기준 불산은 7월 24일 안전재고 아래로 내려갑니다.

### 재고 실사·조정 흐름

```text
시스템 수량  8,000kg
실사 수량    8,120kg
차이          +120kg

조정 사유    [실사 차이 ▼]
증빙 첨부    [파일 선택]
              [조정 요청]
```

사용자가 현재 수량을 직접 편집하는 대신 조정 이벤트를 만들게 합니다.

### 운영과 시나리오 구분

이 두 영역은 색과 언어로도 명확하게 나눍니다.

- 운영 관리: 기존 warm cream 톤, `실제 운영 데이터`
- 시뮬레이션: 보라·청색 계열 강조, `가상 시나리오`
- 시나리오 헤더에 기준 시각을 항상 표시
- 시나리오는 `저장`, 운영 데이터는 `확정`·`실제 입고 처리` 등 다른 동사 사용
- 시나리오에서 운영으로 직접 반영하는 버튼은 제공하지 않음

시나리오를 실제 계획에 적용하려면 `운영 반영 요청`을 만들고 변경 전·후를 검토한 후 승인합니다.

### 실수 방지 UX

- 초안과 확정을 명확하게 구분하고, 초안은 계산에서 제외
- 확정 전 변경 영향 요약과 변경 전·후 비교표 제공
- 과거 시각, 비정상 수량, 창고 용량 초과를 저장 전에 경고
- 확정된 기록은 삭제하지 않고 취소·정정 이벤트로 처리
- 동일 자재·예정일·수량의 중복 등록 가능성 경고
- 여러 행 입력 중 오류가 나도 정상 입력값은 보존

## 트리오 합의안

### 제품 정의

> `운영 관리`는 실제 데이터를 안전하게 생성·확정하는 곳이고, `시뮬레이션`은 그 운영 데이터의 스냅샷을 기준으로 미래를 실험하는 곳이다.

### 정보 구조

```text
FAB System
│
├─ 운영 관리
│  ├─ 오늘의 운영
│  ├─ 입고 계획
│  ├─ 생산 계획
│  ├─ 재고 실사·조정
│  └─ 변경 이력
│
├─ 현황 조회
│  ├─ 대시보드
│  ├─ 재고
│  ├─ 공정 사용량
│  ├─ 창고
│  └─ 위험
│
└─ 시뮬레이션
   ├─ 시나리오 목록
   ├─ 조건 편집
   └─ 운영안 비교
```

### 공통 메타데이터

모든 운영 문서에 다음 값을 공통으로 보관한다.

- 상태: 초안·확정·완료·취소
- 적용 또는 현장 발생 시각
- 시스템 입력 시각
- 입력 출처: 수동·파일·API
- 작성자와 승인자
- 버전
- 변경 사유

### MVP 범위

1. 데이터 상태·버전·변경 이력 규칙 확정
2. 입고 계획 등록·확정·실제 입고 처리
3. 생산계획 등록·확정
4. 재고 실사·조정
5. 변경 이력
6. 오늘의 운영 작업 큐
7. CSV·ERP·센서 연동

첫 구현은 전체 운영 관리를 한번에 만들기보다, **입고 계획 하나를 등록부터 실제 재고·DOH·창고 점유율 반영까지 종단으로 완성하는 것**을 우선한다.

## 미결정 사항

- `ProcessUsage.monthlyQty`를 제품 생산량 당 원단위 모델로 어떻게 이전할지
- 운영 데이터 확정에 승인자를 항상 요구할지, 위험 작업에만 요구할지
- 재고 원장을 기존 `Transaction`으로 확장할지, 통합 `InventoryEvent`를 새로 도입할지
- Lot·Handling Unit·Storage Location을 MVP부터 의무화할지
- 시나리오의 `운영 반영 요청` 승인 흐름을 어느 단계에 도입할지
